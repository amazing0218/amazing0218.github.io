Unicode

Unicode是由Xerox、Apple等软件制造商于1988年组成的统一码联盟发起的非盈利项目，旨在对世界上所有的文字进行统一编码，消除多语言编码不兼容问题，下面是关于Unicode相关术语(引用图说我对Unicode的几点理解)：

  • 代码点(code point): Unicode希望为每个字符分配唯一的一个整数，这个整数就叫代码点。
  • 代码空间(code space): 所有的代码点构成的集合就叫代码空间，Unicode的代码空间一共包含1,114,112个代码点，即0x000000 ~ 0x10FFFF, 目前已经分配大约11w个代码点。

  • 代码平面(code plane): Unicode将代码空间分成了17个代码平面， 每个代码平面包含65536个代码点，即#0~#16, #0平面叫基本多语言平面（Basic Multilingual Plane, BMP）,其他平面叫增补平面（Supplementary Planes），目前大部分增补平面都是空着的。

    下面是这些平面的名字和用途:

    • Plane#0 BMP（Basic Multilingual Plane） 大部分常用的字符都坐落在这个平面内，比如ASCII字符，汉字等。
    • Plane#1 SMP（Supplementary Multilingual Plane） 这个平面定义了一些古老的文字，不常用。
    • Plane#2 SIP（Supplementary Ideographic Plane） 这个平面主要是一些BMP中没有包含汉字。
    • Plane#14 SSP（Supplementary Special-purpose Plane） 这个平面定义了一些非图形字符。
    • Plane#15 SPUA-A（Supplementary Private Use Area A）

    • Plane#16 SPUA-B（Supplementary Private Use Area B）

    • Surrogate Code Points 从0xD800到0xDBFF的1024个代码点是High-surrogate代码点，从0xDC00到0xDFFF的1024个代码点是Low-surrogate代码点。这2048个代码点并不是有效的字符代码点，它们是为UTF编码保留的。一个High-surrogate代码点和一个Low-surrogate代码点组成一个代理对（Surrogate Pair），可以在UTF-16里编码BMP之外的某个代码点（1024^2+65,536=1,114,112）。

  • 码元(code unit): 是指一个已编码的文本中具有最短的比特组合的单元。对于UTF-8来说，码元是8比特长；对于UTF-16来说，码元是16比特长；对于UTF-32来说，码元是32比特长。

事实上，在统一码盟之前，国际标准化组织（ISO）也在做同一件事，并制定了ISO-10646标准，即UCS(Universal Character Set), UCS和Unicode编码方式类似，都是将字符集映射为一个唯一的整数，属于现代编码模型第二层概念，即编码字符集。
Unicode的实现方式不同于编码方式，一个字符的Unicode编码是确定的，但是在实际传输过程中，由于不同系统平台的设计不一致性，以及出于节省空间的目的，对Unicode编码的实现方式有所不同。Unicode的实现方式称为Unicode转换格式（Unicode Transformation Format，简称为UTF），常见的Unicode实现方式包括UTF-8、UTF-16、UTF-32、GB18030等。

UTF-8、UTF-16、UTF-32

UTF-32 是Unicode编码32位的实现方式，即对Unicode每个代码点使用4个字节编码， 编码值与代码点的值相等，码元为4字节。优点：定长编码，方便检索第N个字符；缺点：空间浪费较多，尤其是针对英文文档，且不兼容ASCII编码， 所以实际上基本没有使用。
UTF-16 是Unicode编码16或32位的实现方式，即对Unicode BMP代码平面的每个码点使用2个字节编码，其他增补平面使用4字节编码，属于变长编码， 码元2字节。对于BMP代码平面（0x0000~0xFFFF）的代码点，使用两个字节编码，编码值和码点相等。对于增补平面（0x10000~0x10FFFF）的代码点，需要使用BMP中的代理代码点来表示，其编码算法如下：

  1. 判断代码点值x是否在[0, 0xFFFF]内， 是，使用两个字节编码， 否则第2步；
  2. 代码点值减去0x10000, 得到的值y必定在[0, 0xFFFFF]内， 所以可用20bit编码；
  3. 将y值的高10bit加上0xD800得到高位代理代码点，将y值的低10bit加上0xDC00得低位代理代码点，形成一个代理对，共四个字节。

由于高位代理代码点、低位代理代码点、BMP有效代码点三者互不重叠，所以解码很简单，同时也意味着UTF-16是 自同步(self-synchronizing) 的，即通过检查一个码元，就能断定下一个字符的起始码元位置，而不必重头开始分析，以确定字符码元的边界。UTF-16码元是2字节，有大端（BE）和小端(LE)概念，所以在传输和存储的时候，需要确定字节序，常常会在数据前面加一个BOM(Byte Order Mark), 大端BOM=U+FEFF, 小端BOM=U+FFFE。
UCS-2是UTF-16的子集，即BMP中的字符，UCS-2与UTF-16编码完全一样， 但是USC-2不支持Unicode增补平面的字符。Java中的Unicode编码实现方式，也是使用的UTF-16编码， 并且JSR 204开始对Unicode增补平面的字符支持。
UTF-16优点：大部分常用字符都使用2个字节，相对于UTF-32比较节省空间，而且是自同步，意味传输出错，可以自动恢复，减少信息丢失。缺点：不兼容ASCII编码（老的英文文档用UTF-16解码打不开）, 对于英文字母为主的字符流，还是比较占空间, 还有烦人的字节序问题。
UTF-8 是Unicode编码变长实现方式，与赫夫曼编码类似，也是一种前缀编码（即任意字符的编码，不可能是其他字符编码的一个子串），所以也具有自同步特性。UTF-8最初可以使用1~6个字节为每个字符编码，2003年RFC 3629重新规定，只能使用最多4个字节，具体如下：

  1. 0x00~0x7F,即ASCII字符使用一个字节编码；
  2. 0x80~0x7FF,即拉丁文，希腊文、阿拉伯文等字符，使用两个字节编码；
  3. 0x800~0xFFFF, 即BMP中的其他字符，包括绝大部分中文，使用三个字节编码；
  4. 0x10000~0x10FFFF, 即增补平面字符，使用四个字节编码。
代码码点范围 二进制 UTF-8编码
0x00 ~ 0x7F(128个） 0zzzzzzz 0zzzzzzz
0x80 ~ 0x7FF（1920个） 00000yyy yyzzzzzz 110yyyyy 10zzzzzz
0x800 ~ 0xD7FF 及 0xE000 ~ 0xFFFF(61440个） xxxxyyyy yyzzzzzz 1110xxxx 10yyyyyy 10zzzzzz
0x1000 ~ 0x10FFFF (1048576个） 000wwwxx xxxxyyyy yyzzzzzz 11110www 10xxxxxx 10yyyyyy 10zzzzzz

UTF-8的优点

  1. 兼容ascii编码；
  2. 因为码元为一个字节，所以没有字节序问题；
  3. 对于以英文为主的互联网而言，能大大减少文件大小；

UTF-8的缺点

  1. 变长编码，检索字符不方便；
  2. 对其他字符，如中文字符而言，比较占空间，因此以中文为主的网页，可以选择Unicode的GB18030实现方式。
